# Phase 1 Week 3 Day 14 - Campaign Register Step 2 & 3 Implementation Report

## 개요
브랜드 사용자가 캠페인을 등록할 때 사용하게 될 **멀티스텝 폼의 2단계(원고 가이드)와 3단계(확인 및 등록)**, 그리고 최종 **등록 완료 화면 및 백엔드 로직**을 구현했습니다.

## 주요 구현 사항

### 1. 캠페인 등록 Step 2 (원고 가이드)
- **필수 문구**: 최대 300자까지 입력 가능한 `textarea`를 제공하여 원고 내 필수 요구사항을 기재할 수 있게 구현했습니다.
- **금지어 및 해시태그 (Tag Chip UI)**: 
  - 엔터키 입력 시 배열에 데이터가 추가되는 칩(Chip) 형태의 UI를 적용했습니다.
  - 최대 개수 제약(금지어 20개, 해시태그 30개)을 두었습니다.
  - 해시태그의 경우 입력 시 사용자가 `#`을 붙이지 않더라도 자동으로 `#`이 선행되도록 로직을 추가했습니다.
- **공정위 가이드라인 안내**: 해시태그 입력 필드 상단에 공정위 지침에 따라 `#광고`, `#협찬`이 자동으로 포함된다는 안내 배너 및 시각적 예시를 제공했습니다.
- **참고 URL**: 추가 및 삭제 버튼이 포함된 동적 배열 입력 폼을 제공하였으며, 클라우드 함수에서 정규식을 이용해 형식적 유효성을 검증하도록 처리했습니다.

### 2. 캠페인 등록 Step 3 (요약 및 등록)
- 사용자가 앞서 Step 1과 Step 2에서 입력한 모든 데이터를 섹션별로 읽기 전용 상태로 요약하여 출력합니다.
- 각 섹션 헤더에는 **수정** 버튼을 배치해, 클릭 시 해당 스텝으로 직접 되돌아가서 정보를 수정할 수 있도록 연동했습니다.
- 하단 영역에 수동 임시저장 및 최종 **캠페인 등록** 버튼을 제공하여 제출 전 최종 점검이 가능하게 구성했습니다.

### 3. 클라우드 함수 (`createCampaign.ts`, `validators.ts`)
- **`validateCampaignInput`**: Step 1과 Step 2에서 입력받은 모든 폼 데이터에 대한 필수값 검사 및 타입 무결성, 제한값 범위 검증(예: 참가자 수 1~100, URL 정규식 검사 등)을 수행합니다.
- **`createCampaign` 로직**:
  - 함수 호출 시 사용자의 토큰 정보(`role === 'brand'`)를 기반으로 권한을 검증합니다.
  - 만약 프론트엔드로부터 기존 임시저장 상태인 `campaignId`가 넘어왔다면(즉, Draft에서 이어 작성하는 경우), 중복 생성 방지를 위해 해당 문서를 `status: 'active'` 로 업데이트합니다. 신규일 경우에는 새 문서를 생성합니다.

### 4. 등록 완료 화면 (Step 4)
- 등록 로직이 성공(`isCompleted === true`)할 경우, 멀티스텝 폼 영역을 지우고 체크 아이콘과 함께 "캠페인이 등록되었습니다!" 안내를 렌더링합니다.
- 완료 후 대시보드로 이동하거나, 해당 캠페인의 신청자 관리 페이지(`/brand/campaigns/{id}/applicants`)로 진입할 수 있도록 CTA 버튼을 제공합니다.
