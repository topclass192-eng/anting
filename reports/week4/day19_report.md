# Phase 1 Week 4 Day 19 - 캠페인 상세 및 신청 플로우 보고서

## 개요
인플루언서 사용자가 앤팅(Anting) 캠페인의 상세 정보를 확인하고, 지원 신청(Apply)을 진행할 수 있는 플로우를 성공적으로 구축했습니다. 특히 경쟁이 치열한 캠페인 신청 과정에서 동시성 문제로 모집 정원이 초과되는 것을 막기 위해 백엔드에 **Firestore 트랜잭션**을 적극 활용했습니다.

## 주요 구현 사항

### 1. 프론트엔드 (Apps/Web: `CampaignDetail.tsx`)
- **레이아웃 디자인**: 모바일 환경에 최적화된 앱 라이크(App-like) 화면으로 설계.
  - 뒤로가기 버튼이 포함된 투명한 상단 네비게이션(App Bar).
  - 캠페인의 썸네일을 강조하는 가로 꽉 찬 Header 이미지.
  - 부드러운 UI 연결을 위해 `rounded-t-3xl` 등 둥근 모서리 컨테이너 활용.
- **주요 정보 렌더링**: 캠페인명, 브랜드, 카테고리, 플랫폼 뱃지, 마감일, 모집현황 및 활동지역 표시.
- **원고 가이드라인**:
  - 필수 안내 사항, 텍스트 줄바꿈 유지(`whitespace-pre-wrap`).
  - 해시태그 목록과 공정위 지침 알림 영역 표시.
  - 금지어 강조 표시 (Red 뱃지).
- **하단 고정 지원 (Sticky Bottom CTA)**: 
  - 스크롤을 내려도 항상 하단에 위치하여 사용성이 뛰어난 Primary 버튼 배치.
  - 마감 상태일 경우 버튼을 비활성화하고 "마감된 캠페인"으로 표시.
  - API 통신 중에는 로딩 스피너 애니메이션 처리.
- **결과 모달 및 에러 표시**:
  - 에러 상황 (중복 신청, 인원 초과 등) 시 하단에 빨간색 Alert 알림 박스 노출.
  - 신청 성공 시 중앙에 팝업(모달)을 띄워 시각적인 완료 상태 피드백과 함께 목록으로 복귀할 수 있는 동선 제공.

### 2. 백엔드 (Cloud Functions: `apply.ts`)
- **Firestore 트랜잭션 동시성 제어 (`db.runTransaction`)**:
  - 여러 사용자가 동시에 신청할 경우 발생할 수 있는 초과 모집(Overbooking) 현상을 원천 차단하기 위해 트랜잭션을 도입.
  - `applications` 컬렉션 쿼리(`get`) 후 생성(`set`) 및 `campaigns` 카운트 업데이트(`update`)를 원자적(Atomic)으로 묶어 처리.
- **다중 검증 (Validation)**:
  1. **마감일 검증**: 현재 서버 시간(혹은 함수 실행 시간)과 캠페인의 `deadline`(해당 일 23:59:59 기준)을 비교하여 만료 여부 확인.
  2. **모집 인원 검증**: `currentApplicants >= recruitmentCount` 확인.
  3. **중복 지원 검증**: 해당 `uid`와 `campaignId`의 조합으로 이미 `applications`에 등록된 문서가 존재하는지 점검.
- **에러 응답 최적화**: 위 검증에서 실패할 경우 명시적인 한국어 에러 메시지(예: "이미 신청한 캠페인입니다.")를 던지고, 캐치 구문에서 이를 파싱해 프론트엔드로 안전하게 반환하여 Alert에 표출.
